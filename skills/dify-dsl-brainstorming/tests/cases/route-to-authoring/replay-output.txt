**候选方案**
1. 推荐方案：最小工作流
   - `start` 收 `url`
   - `http-request` 用 `GET {{#start_node.url#}}`
   - `tool/json_parse` 解析 `http_node.body`
   - `end` 输出 `http_node.status_code` 和 `tool_node.result`
   - 收益：最贴合需求，和参考模板一致，验证成本最低。
   - 风险：非 JSON 响应时会在解析节点失败，错误路径较朴素。

2. 增强方案：加显式错误分支
   - 在 HTTP 或解析后增加条件分支，分别处理非 `2xx` 与非 JSON。
   - 收益：运行期行为更可控。
   - 风险：超出当前“最小充分”需求，设计和验收都更重。

3. 替代方案：用代码节点代替 JSON Parse
   - 收益：可自定义解析与报错格式。
   - 风险：没有必要，引入更高复杂度，不如内置工具稳定直接。

**推荐结构与边界**
- 推荐走方案 1。
- 节点职责边界：
  - `start`：只负责收输入。
  - `http-request`：只负责请求目标 URL，并暴露 `status_code/body`。
  - `json_parse`：只负责把响应体转成结构化对象。
  - `end`：只负责输出契约，不承担转换逻辑。

**关键数据流**
- `start_node.url` -> `http_node.url`
- `http_node.body` -> `tool_node.json_string`
- `http_node.status_code` -> `end.status_code`
- `tool_node.result` -> `end.parsed_data`

**错误处理、降级与回退**
- 当前推荐按最小方案处理：
  - HTTP 失败时，保留 `status_code` 作为主要观测字段。
  - 返回体非 JSON 时，由 JSON Parse 节点自然失败，不额外包装。
- 明确不做：
  - 不在本轮加入认证、重试、白名单、复杂异常分支。
  - 不把最小需求升级成“健壮生产版工作流”。

**测试与验收方式**
- 能判断为 `workflow` 模式。
- `start` 节点存在必填 `url` 输入。
- `http-request` 使用 `GET`，目标地址来自输入变量。
- `json_parse` 解析来源是 HTTP 响应体。
- `end` 只输出 `status_code:number` 和 `parsed_data:object`。
- 结构应与参考样本保持同一骨架级别，不额外扩展节点。
