# Dify DSL 设计文档模板

在需要把头脑风暴结果固化成文档时，按这个顺序写，保持简洁但可执行。

## 1. 背景

- 当前问题是什么
- 为什么现在要做
- 现状中最关键的约束

## 2. 目标

- 这次设计必须达成什么
- 怎样算成功

## 3. 任务类型与模式判断

- 这是新建、只读审查，还是修复 / 重构
- 候选模式是 `workflow`、`advanced-chat` 还是 `rag_pipeline`
- 为什么这样判断

## 4. 非目标

- 这轮明确不做什么
- 哪些问题留到后续阶段

## 5. 方案对比

对每条候选方案至少写清：

- 改动范围
- 主要优点
- 主要风险
- 为什么采用或不采用

## 6. 推荐方案

- 总体结构
- 关键模块与职责边界
- 依赖关系

## 7. 数据流 / 交互流

- 主要输入从哪里来
- 中间怎么流转
- 输出如何产生
- 异常路径怎么处理

## 8. 错误处理与降级

- 哪些地方最容易失败
- 失败后如何回退
- 哪些情况必须显式报错

## 9. 测试与验收

- 需要验证哪些主路径
- 需要验证哪些边界或异常路径
- 完成后用什么证据判断设计落地成功

## 10. 待确认问题

- 仍依赖用户决策的点
- 仍依赖外部信息的点

## 11. 下一流程

- 准备进入哪个 skill
- 为什么是它
- 是否仍有非阻塞待定项
