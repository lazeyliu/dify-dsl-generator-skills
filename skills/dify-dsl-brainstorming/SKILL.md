# dify-dsl-brainstorming

把模糊的 Dify DSL 任务收敛成可执行设计，不要把“先聊清楚”偷换成“直接开始生成或改 YAML”。

<强约束>
只要用户主动进入这个 skill，在阻塞未知项清零前，不要生成 DSL、不要修改 DSL、不要切到后续 skill、不要把讨论伪装成实现。
</强约束>

## 工作流

1. 先检查当前项目上下文：关键文件、现有约束、目录结构、最近相关改动。
2. 明确列出已知项、未知项和当前目标是“新建 / 只读审查 / 修复优化重构”中的哪一类。
3. 一次只问一个问题，优先使用容易回答的选项式问题；不要一条消息塞进多个澄清问题。
4. 如果请求明显过大，先拆成更小的 Dify DSL 子问题，只对当前最小可落地问题继续做设计。
5. 在澄清后，给出 2 到 3 条候选方案，先说推荐方案，再说取舍理由。
6. 按复杂度分段呈现设计，并让用户确认：至少覆盖目标、边界、模式判断、节点形态、错误处理、验收标准。
7. 显式维护“未知项台账”，把未知项标成 `阻塞未知项` 或 `非阻塞待定项`。
8. 只有当阻塞未知项被清空，或用户明确接受剩余项只是待定项时，才允许结束头脑风暴。
9. 区分“允许输出文本草稿 / 方案”与“允许落盘修改文件”。如果下一步只是进入 `authoring` 输出 DSL 草稿，而不是直接改文件，不要把“能否落盘”误判成阻塞未知项。
10. 默认自动推荐并路由到下一个 skill，但必须先说明准备进入哪个 skill、为什么；如果用户改路由，服从用户决定。

## 提问原则

- 先问目的、约束、成功标准，再问实现细节。
- 每轮只推进一个决策点，不要并行追问。
- 对已有 DSL，先问“只读判断”还是“允许修改”；不要默认可以改文件。
- 对“生成新 DSL”任务，先问是“输出草稿即可”还是“需要直接落盘修改”；不要把这两者混成同一个权限问题。
- 对已有代码库或 skill 仓库，先顺着现有模式提问；只有当前结构真的妨碍目标时，才把重构纳入设计。
- 如果需求本身互相冲突，先把冲突点说清楚，不要强行拼出折中方案。

## 方案比较原则

- 至少给出 2 条可行路线；复杂任务给 3 条。
- 每条路线都说明：改动范围、收益、主要风险、验证成本。
- 推荐方案必须给理由，不能只报结论。
- 不要把“做得更多”误当成“更好”；优先最小充分方案。

## 未知项门禁

默认把这些问题视为阻塞未知项，除非用户明确接受它们暂缓：

- 目标是新建 DSL、只读审查，还是修复 / 优化 / 重构。
- 目标形态是 `workflow`、`advanced-chat` 还是 `rag_pipeline`。
- 如果任务是修复 / 重构：是否允许实际修改文件，还是只能只读分析。
- 如果任务是新建 DSL：是先输出文本草稿即可，还是要求直接落盘到文件。
- 成功标准是什么，例如“可导入 staging”“只要给最小修复方案”“先选模板不写正文”。
- 是否必须从模板起手，还是允许纯手写骨架。

如果这些问题仍然模糊，不要结束头脑风暴。

特别约束：

- 对新建 DSL 场景，只要“模式、目标、成功标准、模板策略”已经明确，且用户接受先看文本草稿，就可以路由到 `dify-dsl-authoring`。
- 不要因为“尚未确认能否落盘修改文件”而阻止进入 `dify-dsl-authoring`。

## 设计输出最低要求

写设计时，至少覆盖这些部分：

1. 背景与目标
2. 非目标 / 明确不做
3. 候选方案与取舍
4. 推荐结构与职责边界
5. 关键数据流或交互流
6. 错误处理、降级与回退
7. 测试与验收方式
8. 待确认问题

如果需要成文，使用 [references/design-doc-template.md](references/design-doc-template.md) 作为骨架。

## 路由规则

- 从零创建 DSL，或先选模板再生成骨架：
  进入 [../dify-dsl-authoring/SKILL.md](../dify-dsl-authoring/SKILL.md)
- 只读审查、风险分级、导入判断、发布结论：
  进入 [../dify-dsl-review/SKILL.md](../dify-dsl-review/SKILL.md)
- 修复、优化、最小改动、结构重构：
  进入 [../dify-dsl-refactor/SKILL.md](../dify-dsl-refactor/SKILL.md)
- 如果目标仍混合不清、冲突、或存在阻塞未知项：
  继续停留在当前 skill，不要强行路由

## 视觉化原则

只有在“看图明显比读字更容易理解”时，才使用视觉化辅助；并且必须满足两个条件：

1. 当前平台确实提供对应工具。
2. 用户明确同意使用。

如果这两个条件任何一个不满足，就继续保持纯文本头脑风暴。

## 自检标准

- 设计是否回答了“为什么这样做”，而不只是“准备怎么做”。
- 每个模块是否边界清楚，别人不用读内部实现也能知道它负责什么。
- 是否已经明确下一步应该进入哪个 skill，以及为什么。
- 是否已经写清错误路径、回退策略和验收标准。
- 是否混入了用户没要求的功能或重构。
- 是否在阻塞未知项未清零前提前进入实现或下一流程。
