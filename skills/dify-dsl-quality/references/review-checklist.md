# 审核清单

## 通用检查

- 已明确顶层 `kind`。
- 若 `kind: app`，已明确 `app.mode`，且与输出节点类型匹配。
- 若 `kind: rag_pipeline`，已明确 `rag_pipeline` 顶层块与名称、图结构。
- `workflow.graph.nodes` 与 `workflow.graph.edges` 都存在。
- 每个节点都有唯一 `id`。
- 每条边的 `source`、`target` 都能在节点列表里找到。
- 每个节点 `data.type` 与节点文档一致。
- 所有变量选择器都引用了存在的节点、系统变量、对话变量或环境变量。
- 没有把明显的 UI 噪声字段误当成业务必填字段。
- 最终 DSL 已完成 YAML 语法校验、实体/模型级校验和独立复核，或已明确说明缺失项。
- 报告已明确采用的复核模式、参与角色、冲突点和仲裁结果（如适用）。
- 报告已区分“已直接采纳的改进”“有条件但未执行项”“待确认项”。

## 变更影响面

- 若修改了既有 DSL，已列出节点、selector、变量、输出约定、fallback、依赖、成本的变化。
- 已区分“预期内变化 / 非预期变化 / 待确认变化”。
- 若不存在基线，报告中已明确写出“无可比基线”。

## 输出闭环检查

- `workflow` 至少有一个 `end`。
- `advanced-chat` 至少有一个 `answer`，或明确说明为何只输出 `end`。
- `rag_pipeline` 至少有一条主路径到用户声明的最终处理节点。
- 每个终点节点都能从入口节点走到。
- 关键终点不是只依赖“成功输出”单一路径。

## 容器节点检查

- `iteration` 与 `loop` 都显式写了 `start_node_id`。
- `iteration-start` / `loop-start` 节点真实存在，且位于容器内部。
- 容器内部节点和边没有引用容器外不存在的临时 ID。
- `output_selector`、`break_conditions`、`loop_variables`、`items` 等字段完整。

## 提示词审查

- 所有带提示词的节点都检查过精炼性，避免明显重复背景、重复限制或低价值长段铺垫。
- 角色、任务、输入、输出、语言要求和边界条件清楚，不依赖模型自行猜测。
- 没有互相冲突的指令。
- 变量、占位符、选择器和上下文字段引用准确，且不会把未解析模板暴露给最终用户。
- 输出格式、链接生成条件、空结果处理方式和失败兜底写清楚。
- 提示词与业务目标一致，不会在关键信息缺失时做不合理推断。
- 提示词不会把未验证事实写成确定事实，不会输出明显夸大或误分类结果。
- 多语言场景下，各语言 prompt 的约束与禁止项保持一致。

## 专项审查

- 已检查数据约定与变量语义，确认上游输出含义、空值语义与下游假设一致。
- 已检查状态污染与清理边界，确认不会把旧会话结果误带入新判断。
- 已检查关键动作前置条件，例如生成链接、写入关键变量、进入关键分支前是否拿到了必要信息。
- 已检查决策覆盖与不可达分支，确认默认分支不过宽、关键分支不是死分支。
- 已检查证据来源与可追溯性，关键结论能回答“来自哪个上游变量、哪个工具结果或哪个证据快照”。
- 已检查多轮一致性与时效性，避免跨轮引用漂移或把时效性信息当成稳定事实。
- 已检查失败恢复与状态回退，确认异常或空结果后不会误保留旧状态、误走成功路径，必要时支持轻量重试或备用链路。
- 已检查副作用与幂等性，确认外部调用、写状态动作和重试行为不会造成重复写入或不可控后果。
- 已检查资源边界与终止性，确认循环、分页、聚合和长链路有明确的上限、超时或早停条件。
- 已检查回退路径约定一致性，确认主路径和 fallback 路径的输出字段、类型与语义一致，或已加归一化层。
- 已检查敏感信息与最小暴露，确认 prompt、日志、错误信息、answer 和工具参数没有透传不必要敏感字段。
- 已检查依赖与版本假设，确认插件、工具、字段写法和外部返回结构的兼容前提已明确。
- 已检查可观测性与成本时延，关键决策可解释，且不存在明显无效调用。
- 已检查编排精炼度与职责边界，避免一个节点同时承担过多理解、决策和呈现职责。

## 观测字段约定

- 关键决策节点已输出适用的 `reason_code`、`selected_path`、`evidence_ref`、`fallback_reason`。
- 如适用，已输出 `contract_version`、`normalization_notes`、`confidence_level`。
- 观测字段默认用于内部，不向终端用户暴露敏感信息。

## 能力块边界约定

- 每个能力块至少声明了输入、输出、前置条件、副作用、失败语义、fallback、成本等级、决定性等级。
- 能力块与策略层、文案层、渲染层边界清楚。
- 场景包只组合能力块，不复制能力块内部逻辑。

## 提精准度与提灵活度

- 高风险业务动作已尽量改成规则护栏，而不是只靠 prompt 约束。
- 已定义关键中间对象的最小 schema，避免节点之间长期传递语义模糊的自由文本。
- 已做输入归一化与证据快照，关键决策和最终生成基于统一清洗后的输入与冻结证据。
- 已明确决定性边界，知道哪些逻辑必须规则化，哪些逻辑允许交给模型生成。
- 复杂链路已优先拆成“抽取 / 决策 / 呈现”或其他中间语义层，避免一个节点包办所有逻辑。
- 若任务允许定义低置信度策略，已考虑置信度 / 拒答门与证据约束；否则已在报告中说明未启用原因。
- 若存在样例集、测试入口或回放输入，已执行黄金样例与对抗样例回放；否则已在报告中列为优先补齐项。
- 若存在稳定可复用的轻量重试、备用解析链或安全 fallback，已设计运行期自修复；否则已在报告中说明未启用原因。
- 若链路存在稳定的轻重两档，已设计能力裁剪；否则已在报告中说明为什么不适合裁剪。
- 若业务明确需要多种输出形态，已支持输出形态切换；否则已在报告中说明当前只保留单一输出的原因。
- 已输出覆盖率矩阵，明确主路径、fallback、空结果、异常、低置信度、多轮状态和降级路径是否覆盖。
- 已判断是否已经够用，明确当前结构是过重、合理还是过轻。
- 已定义升级条件，明确关键证据缺失、证据冲突、高风险动作和上线前检查未通过时的动作。
- 重要问题已带稳定 code，而不是只写自然语言描述。

## 独立复核模式

- 高复杂度、分工清晰且上下文相对独立的问题，优先异步并行复核。
- 前一轮结论会明显影响后一轮判断的问题，优先同步串行复核。
- 如果只有一个最主要风险面，允许只启用 1 个只读子代理，但要在报告中说明为何没有多方复核。
- 每个子代理都应有独立提示词，不要把所有检查项都塞进一个通用提示词后再简单改角色名。
- 如果多方复核结论冲突，已明确列出冲突点、仲裁结果和仍待确认项，而不是直接拼接多个互相矛盾的结论。

## 上线前检查

- 结构检查、输入输出约定检查、风险检查、回归检查、观测字段检查、复核一致性检查已逐项给出通过 / 失败 / 待定 / 豁免状态。
- 如果没有样例集或回放入口，回归检查已标记为“待定”而不是“通过”。
- 如果使用豁免，豁免理由已写清。
- 修改既有 DSL 时，如果存在旧 DSL 或旧基线，已做变更影响面与回归差异审查；如果不存在，已明确写出“无可比基线”。

## 审核结论

- 可直接导入
- 需要人工确认
- 明显不完整

给结论时必须写原因，不能只写标签。
如果 3 轮检查没有全部完成，不应表述为“已精准校验”。
不要用 `YAML_OK`、`STRUCTURE_OK`、`PY_CODE_OK` 这类裸状态码代替审核说明；如果保留，只能作为附注，不能当主结果。
若存在任一阻塞项，结论至少应为“明显不完整”。
若没有阻塞项，但仍有高风险项、关键检查项失败或关键假设未验证，结论至少应为“需要人工确认”。
