# 子代理复核编排手册

这份手册只回答一个问题：这轮复核到底该怎么组织。

进入本页前，先用 [mode-selection-matrix.md](mode-selection-matrix.md) 选复核模式。

## 默认编排

如果平台能力、会话策略和用户授权都允许，默认用这一顺序：

1. 并行启动 3 个只读复核器
2. 汇总三方问题
3. 判断是否存在真正冲突
4. 只有存在冲突，或任务本身已经明确要求交付判断时，再启用 `上线检查复核器`
5. 输出统一结论

## 角色顺序

默认前三方：

- `字段约束复核器`
- `链路闭环复核器`
- `提示词与输入输出约定复核器`

按需启用：

- `上线检查复核器`

## 什么时候启用上线检查复核器

只在这几种情况启用：

- 三方结论对“能否交付 / 能否导入 / 风险是否阻塞”存在真实冲突
- 用户明确要求发布门禁、覆盖率缺口、观测字段或上线前检查结论
- 当前任务已经不是单纯的只读审查，而是需要归并成统一交付结论

这些情况不要启用：

- 只是想做普通只读审查
- 三方没有实质冲突，只是措辞不同
- 当前还没拿到前三方基础结论

## 退化顺序

如果当前环境不能满足默认编排，按这个顺序退化：

1. `3 个只读子代理`
2. `1 个只读子代理`
3. `0 个子代理`

### 单子代理

只在下面场景使用：

- 任务范围很小
- 风险面高度单一
- 平台成本或上下文限制不适合多方复核

这时要明确说明为什么没有使用多方复核。

### 无子代理

只在下面场景使用：

- 平台不支持子代理
- 当前会话策略不允许
- 用户明确禁止

这时必须：

- 执行完整自检
- 明确写出 `未做独立复核`
- 把结论至少提升到 `需要人工确认`

## 结果汇总规则

- 先保留三方原始问题，不要先做“平均化结论”
- 只把真正互相冲突的项列入仲裁
- 仲裁结果要写清：
  - 采纳哪一方
  - 为什么采纳
  - 另一方为什么不采纳
- 如果证据不足，不要强行拍板，保留为 `待确认项`

## authoring / review / refactor / governance 的接法

- `authoring`
  先生成草稿，再进入本编排 skill 做独立复核
- `review`
  当任务复杂到需要正式多方复核时，转到这里
- `refactor`
  修改后如果需要独立复核，再转到这里
- `governance`
  如果用户明确要求多方独立复核或冲突归并，再先转到这里组织复核；否则可以直接承担交付判断入口。
