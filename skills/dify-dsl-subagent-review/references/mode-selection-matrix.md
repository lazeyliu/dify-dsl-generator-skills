# 复核模式决策表

先选模式，再派发角色。不要一上来默认并行。

## 决策表

| 当前信号 | 推荐模式 | 原因 |
| --- | --- | --- |
| 字段、图结构、提示词与输入输出约定可以相对独立检查，且任务同时覆盖多个风险面 | 异步并行 | 先并发拿到多方意见，速度最快，也最适合复杂样本 |
| 后一轮判断明显依赖前一轮结果，例如必须先确认结构站得住，再决定是否继续审提示词、门禁或归并 | 同步串行 | 避免在错误前提上放大后续判断 |
| 风险面高度单一，例如几乎只剩图闭环、selector 或某一个局部问题 | 单子代理 | 不要为了形式上的“多方复核”增加噪音 |
| 平台不支持、会话不允许或用户明确禁止自动拉起子代理 | 无子代理 | 只能退化为完整自检，并明确写出 `未做独立复核` |

## 快速判断

### 选 `异步并行`

满足下面任意 2 条，优先并行：

- 同时涉及字段、链路、提示词或交付门禁多个面向
- 样本复杂度中高
- 任务目标本身就是“多方独立复核”
- 用户显式要更快拿到多方意见

### 选 `同步串行`

满足下面任意 1 条，优先串行：

- 你需要先确认结构或字段是否成立，再决定后面的复核有没有意义
- 后续结论高度依赖前一轮结果
- 样本不大，但检查之间有明显前后依赖

### 选 `单子代理`

满足下面全部条件，可以单子代理：

- 风险面很窄
- 主要问题集中在一个角色的职责范围里
- 额外复核器大概率只会重复同一个结论

### 选 `无子代理`

只在没有自动子代理能力或没有授权时使用，不是为了省事。

## 样例映射

- `knowledge-retrieval-chatflow.yml`
  默认更适合 `异步并行`
- `min-advanced-chat.yml`
  如果目标是“先确认结构，再继续看提示词与门禁”，适合 `同步串行`
- `broken-selector.yml`
  如果目标只是确认图闭环和 selector 悬空，适合 `单子代理`

## 选择后必须写出的信息

- 选择了哪种模式
- 为什么不是另外两种更接近的模式
- 实际启用了几个角色
- 如果发生退化，退化原因是什么
