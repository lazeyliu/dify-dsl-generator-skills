# 字段说明规范

节点文档里的字段不应只列字段名，还要说明字段用途。

## 必填字段说明方式

每个必填字段至少说明一件事:

- 这个字段控制什么
- 缺失后会造成什么问题
- 它通常引用什么数据

推荐写法:

- `type`: 指定节点类型，决定当前节点按哪种语义执行。
- `model`: 指定模型配置，没有它就无法发起模型调用。
- `output_selector`: 指定容器向外暴露哪个内部结果，缺失后容器结果无法被下游消费。

## 选填字段说明方式

每个选填字段至少说明一件事:

- 什么时候建议填写
- 不填写时的默认行为
- 填写后会影响哪段链路

推荐写法:

- `memory`: 需要带上下文推理时再启用。
- `vision`: 需要图像输入时再补齐。
- `timeout`: 对外部请求或人工输入有超时要求时填写。

## 字段说明优先级

优先解释这些字段:

- 会改变执行路径的字段
- 会改变输入输出结构的字段
- 会影响上下游贯通的字段
- 容器节点的关键字段
- 触发器入口字段

## 不必过度解释的字段

下面这类字段通常只需要一句话:

- `title`
- `desc`
- `version`
- 一般性的 UI 辅助字段

## 贯通导向写法

字段说明最好和链路一起写，不要脱离上下游单独解释。

推荐写法:

- `query_variable_selector`: 指定检索节点读取哪个查询变量，它决定上游输入是否能真正进入检索链路。
- `index_chunk_variable_selector`: 指定索引节点消费哪一段分块结果，它决定文档处理链路是否能落到知识库。

## 上下游写法

每个节点文档都要显式写:

- 可承接上游节点
- 可衔接下游节点

并统一拆成三级:

- `推荐`
- `可用但需人工确认`
- `不推荐`

写法要求:

- `推荐`: 常规编排中稳定、自然、风险低的连接方式。
- `可用但需人工确认`: 理论上可接，但容易受字段形态、执行时机或业务语义影响。
- `不推荐`: 容易造成语义混乱、双入口、死链或不必要复杂度的连接方式。
