# 前向验证手册

在重大结构调整、路由重构、验证资产变化或准备宣称“可直接导入 / 已精准校验”前，用这份手册做只读前向验证。

## 核心原则

- 用真实样本和真实任务表述，不用学术化提问。
- 让子代理像在处理用户任务，不要告诉它“你正在测试 skill”。
- 只给最小必要上下文，不泄漏预期答案、已知缺陷和打算采用的修复方案。
- 一次只验证一个主要目标：路由是否正确、审查是否到位、模板路线是否命中、还是交付判断是否站得住。
- 当前仓库各 skill 都禁止隐式触发，所以前向验证提示词默认显式点名目标 skill。

## 什么时候做

- 刚重构 `SKILL.md`、目录入口或关键 reference 之后。
- 新增或调整子代理复核规则之后。
- `fast_test_dsl.py` 或 `fast_test_suite.py` 显示入口覆盖缺口之后。
- 准备把 skill 交付给别人复用之前。

## 最小工作流

1. 先用 `python3 skills/dify-dsl-foundations/scripts/fast_test_dsl.py <sample.yml>` 判断这个样本会命中哪些入口 skill。
2. 明确这轮要验证的目标，例如“advanced-chat 多轮状态审查是否到位”。
3. 为每个用例准备最小工件目录：`target.yml`、可选的 `baseline.yml`、`prompt.txt`、`oracle.json`。
4. 选一个全新线程的只读子代理，给它自然用户式提示词。
5. 记录结果：命中的路线、是否提前偏题、是否漏掉中间产物、结论是否过强。
6. 如果失败，改 skill 或参考资料；不要先改提示词把结果“哄对”。

## 用例目录建议

建议每个前向验证用例独立成一个目录，例如：

```text
case-001/
├── prompt.txt
├── oracle.json
├── target.yml
└── baseline.yml   # 可选
```

说明：

- `prompt.txt`：给主线程或子代理的真实任务表述。
- `oracle.json`：只供断言脚本或人工对照使用，不要把里面的预期发现泄漏给被测线程。
- `target.yml`：真实样本。
- `baseline.yml`：只有在做变更影响或回归差异时才需要。

可以用 `python3 skills/dify-dsl-forward-testing/scripts/init_forward_test_case.py <case-dir> --target <sample.yml>` 生成骨架。
仓库内现成用例可放在各自 skill 的 `tests/cases/`，并用 `python3 skills/dify-dsl-forward-testing/scripts/check_forward_test_cases.py <cases-root> [repo-root]` 汇总覆盖。

## 推荐角色名

对外汇报时优先使用有意义的中文角色名，不用系统自动昵称。

- `字段约束复核器`
- `链路闭环复核器`
- `提示词与输入输出约定复核器`
- `上线检查复核器`
- `模板路线验证器`
- `总入口路由验证器`

## Prompt 形状

保持“像真实用户发任务”，不要泄漏诊断。

### 结构审核型

```text
使用 $dify-dsl-review 执行这次只读前向验证。

工作目录: <case_dir>
目标文件: <target.yml>
旧基线: <baseline.yml | 无>

这是一个真实审查任务，不是问答。该 DSL 计划在今天导入 staging。
约束：只读；不要修改文件；不要生成新 DSL。
如果当前平台 / 会话 / 授权策略不允许自动拉起子代理，明确写“未做独立复核”。

请直接行动：
1. 先给出模式判断、节点清单、连边清单和字段检查表。
2. 如果授权允许，启动 3 个只读子代理：
   字段约束检查
   链路闭环检查
   提示词与输入输出约定检查
3. 只有当前三方结论冲突，或存在旧基线需要仲裁时，才启动上线前检查做归并。
4. 输出先列问题，再给结论；问题按阻塞项 / 高风险项 / 中风险项 / 优化项排序；每条问题都带节点 ID 或字段路径。
5. 最终只给一个结论：可直接导入 / 需要人工确认 / 明显不完整。
```

### 修复策略型

```text
使用 $dify-dsl-refactor 分析 /path/to/sample.yml 为什么容易出错。
不要直接重写整份 DSL，先给最小修复方案、关键风险和仍待确认项。不要修改文件。
```

### 模板路线型

```text
使用 $dify-dsl-templates 为这个需求选择最接近的 Dify 模板路线：
“<把真实需求写在这里>”
先说明会进入哪些底座 skill，再给模板、变体和理由。不要修改文件。
```

### 交付判断型

```text
使用 $dify-dsl-review 只读评估 /path/to/sample.yml。
重点回答上线前检查、观测字段约定、覆盖率缺口和最终结论，不要修改文件。
```

## 不要泄漏的信息

- 不要说“我们怀疑它漏读了 templates skill”。
- 不要说“正确答案应该命中 quality 和 governance skill”。
- 不要说“这个样本的已知问题是某个 selector 断了”。
- 不要把你已经写好的审查结论塞给子代理再让它重复。
- 不要把 `oracle.json` 里的预期入口、预期发现或通过条件写回 `prompt.txt`。
- 不要把 `fixture-index` 中的样例名称说成“仓库里一定存在的本地样本文件”。

## 失败信号

- 没有先做模式判断就直接输出大段 YAML。
- 明明是只读审核任务，却开始建议改文件且跳过中间产物。
- 该命中 `quality / governance` 的样本，只做了节点罗列，没有风险分级和门禁结论。
- 明明是模板路线任务，却没有进入 `dify-dsl-templates`。
- 结论过强，例如在未做 3 轮检查时宣称“已精准校验”。

## 建议记录字段

- 样本路径
- 任务提示词
- 目标入口
- 实际命中入口
- 成功 / 失败
- 主要失败信号
- 对应要改的文件
- 是否全新线程
- 是否使用 3 / 1 / 0 个只读子代理
