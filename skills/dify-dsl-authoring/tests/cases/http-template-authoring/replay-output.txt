**模式判断**
这是 `workflow` 模式，不是 `advanced-chat` 或 `rag_pipeline`。原因是目标是接收一个输入参数 `url`，执行 HTTP 请求，做 JSON 解析，并以结构化字段 `status_code` 与 `parsed_data` 结束输出。

**模板选择**
选择 `HTTP 处理链模板`，骨架为 `start -> http-request -> tool/code -> end`。  
选择理由：它与“输入 URL -> 请求外部接口 -> 解析 JSON -> 输出结构化结果”的链路完全一致。  
证据等级：已由样例索引直接支撑；本仓库内还有直接对应的参考样本 [template-http-workflow.yml](/Users/lazeyliu/Downloads/tee/tests/fixtures/dsl/template-http-workflow.yml)。  

**节点清单**
1. `start_node`
2. `http_node`
3. `tool_node`
4. `end_node`

**连边清单**
1. `start_node -> http_node`
2. `http_node -> tool_node`
3. `tool_node -> end_node`

**字段检查表**
- `http_node.data.url` 应引用 `{{#start_node.url#}}`
- `tool_node.tool_parameters.json_string` 应引用 `{{#http_node.body#}}`
- `end_node.outputs[0]` 为 `status_code`
- `end_node.outputs[1]` 为 `parsed_data`

**DSL 草稿**
```yaml
kind: app
version: 0.6.0
app:
  mode: workflow
workflow:
  graph:
    nodes:
      - id: start_node
        data:
          type: start
      - id: http_node
        data:
          type: http-request
      - id: tool_node
        data:
          type: tool
      - id: end_node
        data:
          type: end
          outputs:
            - variable: status_code
            - variable: parsed_data
```
