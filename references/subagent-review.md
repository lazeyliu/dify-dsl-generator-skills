# Subagent 复核

生成 DSL 后，必须做独立复核。只要当前环境支持 subagent，就默认自动启用 3 个只读 subagent 做交叉检查，不需要用户再单独要求。

## 复核模式

- 异步并行
  适合字段、图结构、提示词与风险结论可相对独立审查的情况。默认优先使用这一模式，先并发拿到多方意见，再汇总冲突点。
- 同步串行
  适合后一轮判断明显依赖前一轮结论的情况，例如先确认结构是否可靠，再基于稳定结构检查提示词与策略风险。
- 单 subagent
  适合任务范围很小、风险面单一，或当前环境不适合拉起多方复核的情况。
- 无 subagent
  仅在当前环境没有 subagent 能力时使用，退化为完整自检，并把风险级别至少提升到“需要人工确认”。

推荐分工:

- subagent A: 节点字段与实体约束复核
- subagent B: 连边、选择器、容器闭环复核
- subagent C: 兼容字段、历史字段、提示词、数据契约、安全与可导入性结论复核
- subagent D: 变更影响、观测契约、发布门禁与冲突仲裁复核（按需启用）

如果当前执行环境无法满足 3 个 subagent，自动按下面顺序退化:

- 3 个只读 subagent
- 1 个只读 subagent
- 无 subagent

如果当前执行环境没有 subagent 能力，退化为:

- 完整执行自检清单
- 在报告中明确标记“未做独立复核”
- 把风险级别至少提升到“需要人工确认”

## 复核目标

- 核对节点类型是否覆盖正确。
- 核对每个已用节点的必填字段是否齐全。
- 核对变量选择器、边、容器起点和输出节点是否闭合。
- 标记“强约束”和“兼容字段”的边界。

## 独立提示词

### subagent A: 字段与实体约束复核

```text
请只读审查这份 Dify DSL 设计/草稿，不要修改文件。你只关注节点字段与实体约束：
1. 节点类型是否误用、遗漏或与当前模式不匹配；
2. 每个已使用节点缺少了哪些必填字段；
3. 是否存在字段值域、字段存在性、字段类型、字段形状、字段组合关系或变量类型兼容性问题；
4. `code_language` 与代码内容是否一致，是否存在明显运行时语法风险；
5. 是否存在依赖与版本假设不清、回退契约不一致或输出契约不稳定问题；
6. 给出问题清单，并按阻塞项、高风险项、中风险项、优化项分级。
```

### subagent B: 图结构与选择器闭环复核

```text
请只读审查这份 Dify DSL 设计/草稿，不要修改文件。你只关注图结构、选择器与链路闭环：
1. 是否存在悬空的 edge、variable_selector、value_selector 或 output_selector；
2. 容器节点、内部起点、输出节点和终点路径是否闭合；
3. 是否存在不可达分支、默认分支过宽或关键分支缺失；
4. 前置条件、失败恢复、状态回退是否足以避免误走成功路径；
5. 循环、分页、聚合和长链路是否有明确的资源边界、终止条件和输出上限；
6. 给出问题清单，并按阻塞项、高风险项、中风险项、优化项分级。
```

### subagent C: 提示词、数据契约与风险结论复核

```text
请只读审查这份 Dify DSL 设计/草稿，不要修改文件。你只关注提示词、数据契约、安全与可导入性结论：
1. 带提示词的节点是否存在冗余、冲突、变量引用错误、证据约束缺失、业务不合理、结果不准确或安全注入风险；
2. 数据契约、状态污染、多轮一致性、时效性口径是否清楚；
3. 是否明确了内部语义对象最小 schema、输入归一化、证据快照和决定性边界；
4. 是否存在副作用、幂等性、敏感信息最小暴露或关键动作最小暴露问题；
5. 对黄金样例与对抗回放、置信度门与拒答门、运行期自修复、能力裁剪、输出形态切换这 5 项，逐项判断条件是否满足、是否已执行或已设计、未执行原因是否成立；
6. 给出“可直接导入 / 需要人工确认 / 明显不完整”的结论和理由；
7. 再补 1 到 3 个最高收益改进建议，并区分哪些可直接做、哪些有条件做、哪些待确认。
```

### subagent D: 变更影响、观测契约、发布门禁与结论归并复核

```text
请只读审查当前 DSL 与 A/B/C 复核结论，不要重新做全量审查。你只关注变更影响、观测契约、发布门禁和最终结论归并：
1. 如果存在旧 DSL 或旧基线，列出节点、selector、变量、输出契约、fallback、依赖和成本的关键变化；
2. 检查关键决策节点是否具备最小观测字段，且没有敏感信息泄露；
3. 检查结构门、契约门、风险门、回归门、观测门、复核一致性门的状态；
4. 列出各角色结论之间真正冲突的点，不要把表述差异误判为冲突；
5. 对每个冲突点，判断应采纳哪一方、为什么采纳、另一方为什么不采纳；
6. 如果现有证据不足以仲裁，明确标记为“待确认项”，不要强行拍板；
7. 输出最终统一结论，并保留阻塞项、高风险项、中风险项、优化项分级。
```

## 通用汇总提示词

```text
请只读审查这份 Dify DSL 设计/草稿，不要修改文件。请基于当前 skill 目录中的节点文档与现有实现约束，回答：
1. 节点类型是否完整，是否有误用或遗漏；
2. 每个已使用节点缺少了哪些必填字段；
3. 是否存在悬空的 variable_selector / value_selector / edge；
4. 是否混入了明显不可靠的历史字段或 UI 字段；
5. 是否存在数据契约、状态污染、前置条件、决策覆盖、多轮一致性、失败恢复或安全注入风险；
6. 是否存在副作用与幂等性、资源边界与终止性、回退契约一致性、敏感信息最小暴露、依赖与版本假设问题；
7. 带提示词的节点是否存在冗余、冲突、变量引用错误、证据约束缺失、业务不合理、结果不准确或编排职责混乱风险；
8. 是否明确了内部语义对象最小 schema、输入归一化、证据快照和决定性边界；
9. 对黄金样例与对抗回放、置信度门与拒答门、运行期自修复、能力裁剪、输出形态切换这 5 项，逐项判断条件是否满足、是否已执行或已设计、未执行原因是否成立；
10. 如果存在旧 DSL 或旧基线，列出变更影响面与回归差异；如果不存在，明确写无可比基线；
11. 检查关键决策节点是否具备最小观测字段、能力块是否具备最小契约、发布门禁是否逐项成立；
12. 如果多方复核结论冲突，列出冲突点、仲裁建议和仍待确认项；
13. 给出“可直接导入 / 需要人工确认 / 明显不完整”的结论和理由，并把问题按阻塞项、高风险项、中风险项、优化项分级；再补 1 到 3 个最高收益改进建议，并区分哪些可直接做、哪些有条件做、哪些待确认。
```

## 复核输出要求

- 先列问题，再给结论。
- 问题按严重度排序。
- 每条问题都要指出对应节点 ID 或字段路径。
- 报告里要说明本轮采用了异步并行、同步串行、单 subagent 还是无 subagent 模式。
- 对提示词问题，优先指出具体变量、具体指令冲突或具体业务风险点，不要只写“prompt 可以优化”。
- 对数据契约或状态问题，优先指出具体变量、具体前置条件或具体跨轮污染点，不要只写“需要关注上下文”。
- 对安全或编排问题，优先指出具体输入面、工具面、链接拼接点或职责混乱点，不要只写“有风险”。
- 对副作用、幂等性、资源边界、回退契约和依赖版本问题，优先指出具体节点、具体上限、具体契约或具体版本假设，不要只写“建议优化”。
- 对这 5 类条件项，优先写清“条件是否满足、当前是否已做、未做原因”，不要只写“建议补充”。
- 对建议项，优先明确“可直接做 / 有条件做 / 需确认后做”，不要把三类建议混在一起。
- 如果使用多方复核且结论冲突，要单独输出冲突点、仲裁结果和仍待确认项。
- 如果没发现问题，也要说明剩余风险，例如“依赖兼容字段”“未在真实导入环境验证”。
- 报告中要写清这轮到底完成了 3 个 subagent、1 个 subagent，还是 0 个 subagent。
