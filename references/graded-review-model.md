# 分级审查模型

用于把“校验通过/失败”进一步转成更适合人阅读和决策的审查结论。它不替代 YAML、实体或代码层面的硬校验，而是把这些结果与提示词、业务、输出质量层面的判断合并成一份可执行的审核意见。

## 1. 审查等级

- 阻塞项
  当前不建议导入，或导入后大概率无法正常运行、无法稳定输出或会直接产出错误结果。
- 高风险项
  不一定阻塞导入，但会显著影响正确性、业务安全性、用户体验或人工维护成本。
- 中风险项
  不影响主链路启动，但会影响稳定性、准确性、一致性或后续维护。
- 优化项
  主要影响可读性、冗余度、提示词长度、命名整洁度或报告质量。

## 2. 默认审查维度

- 可解析性与可导入性
- 图结构与选择器闭环
- 运行时代码与返回结构
- 提示词质量与可执行性
- 业务合理性与结果准确性
- 兼容写法与人工确认项
- 数据契约与变量语义
- 状态污染与清理边界
- 前置条件与动作护栏
- 决策覆盖与不可达分支
- 证据来源与可追溯性
- 多轮一致性与时效性
- 失败恢复与状态回退
- 副作用与幂等性
- 资源边界与终止性
- 回退契约与输出一致性
- 敏感信息与最小暴露
- 依赖与版本假设
- 变更影响面与回归差异
- 观测契约
- 能力块契约
- 发布门禁
- 可观测性
- 安全与注入
- 成本与时延
- 编排精炼度与职责边界
- 内部语义对象最小 schema
- 输入归一化与证据快照
- 决定性边界
- 复核冲突仲裁

## 3. 提示词专项审查

带提示词的节点至少覆盖以下项目：

- 精炼性
  是否存在明显重复背景、重复限制、重复示例、低价值解释。
- 清晰性
  角色、任务、输入、输出、语言要求和边界条件是否明确。
- 可执行性
  模型是否容易稳定执行，不需要靠猜测补全隐含规则。
- 指令一致性
  是否存在互相冲突的要求，例如一边要求简洁，一边要求输出长篇固定结构。
- 变量准确性
  变量、占位符、选择器和上下文字段是否真实存在，名称与含义是否匹配。
- 输出约束完整性
  格式、字段、链接生成条件、缺失信息处理方式是否写清楚。
- 业务合理性
  在信息缺失、解析失败、工具失败或上下文不完整时，是否仍能给出安全、合理的结果。
- 结果准确性
  是否存在明显夸大、误导、错链、错分类、未验证即断言的表达。
- 多语言一致性
  多语言 prompt 的约束、结论和禁止项是否一致。
- 安全与注入
  是否容易被用户输入、工具返回、知识片段或未转义变量带偏，是否缺少必要护栏。
- 证据约束
  是否明确限制模型只依据哪些上游字段、工具结果或候选集决策，缺证据时是否会自动回退而不是脑补。

## 4. 典型分级示例

- 阻塞项
  - YAML 无法解析。
  - 节点引用了不存在的 `source`、`target` 或 selector。
  - `code_language: python3`，但代码存在 `SyntaxError`。
  - 提示词引用了不存在的变量，且该变量会直接出现在用户最终回复里。
- 高风险项
  - 未拿到关键资源标识，仍然拼接结果链接或执行关键动作。
  - 提示词同时要求“严格只基于上游变量”与“可自行补全缺失信息”。
  - 失败路径缺失，工具异常后仍继续生成确定性结论。
- 中风险项
  - 提示词约束存在重复，主次不清。
  - 多语言版本表达不完全一致，可能导致分类或输出风格偏移。
  - 兜底文案存在，但解释过弱，用户不易理解为什么未命中。
- 优化项
  - 提示词过长，但没有直接改变结论。
  - 审核报告语句偏机械，可读性一般。
  - 节点命名或字段说明不够统一。

## 5. 结论映射

- 可直接导入
  没有阻塞项，且没有未关闭的高风险项；剩余问题最多为中风险项或优化项。
- 需要人工确认
  没有阻塞项，但仍有高风险项、关键假设未验证，或存在需要业务方确认的准确性问题。
- 明显不完整
  存在任一阻塞项，或核心链路、核心提示词、关键变量引用存在明显错误。

## 6. 输出要求

分级审查输出至少包含：

| 审查维度 | 等级 | 是否影响导入 | 主要发现 | 建议动作 |
| --- | --- | --- | --- | --- |
| 例如: 提示词质量 | 高风险项 | 否，但建议人工确认 | 关键标识缺失时仍会生成结果链接 | 增加缺失信息兜底，并禁止输出占位链接 |

不要只写“Prompt OK”或“Review Passed”。要把问题、影响和建议动作说清楚。
除了审查结论，建议再给 1 到 3 个最高收益的改进动作，不要只停留在挑错。
建议动作要标清“可直接做 / 有条件做 / 需确认后做”。

## 7. 提精准度与提灵活度抓手

- 提精准度优先
  中间语义层、内部语义对象最小 schema、抽取/决策/呈现分层、输入归一化、证据快照、置信度门、证据约束、关键动作规则护栏、决定性边界。
- 提稳定性优先
  黄金样例与对抗样例回放、失败回退、运行期自修复、状态清理和前置条件校验。
- 提灵活度优先
  模板分层、能力块化编排、策略档位、可插拔 fallback、prompt 组件化、输出形态切换、场景包。
- 提兼容性优先
  selector 别名层、适配层、可替换工具输出契约。
- 提安全性优先
  prompt injection 护栏、工具返回过滤、变量转义与关键链接拼接保护。
- 提成本效率优先
  能力裁剪、去掉重复 LLM 调用、把可规则化逻辑从模型中拿出来。

## 8. 落地优先级

### 默认直接做

- 数据契约审查
  做法：对关键变量至少写清来源、类型、空值语义、单位或枚举口径、下游消费者。
- 状态污染审查
  做法：逐个检查会话变量的写入、覆写、清空、失败后回退和跨轮复用边界。
- 前置条件审查
  做法：对生成链接、写关键变量、调用工具、进入关键分支等动作，显式列出必需输入；缺失时改走澄清、回退或拒答。
- 规则护栏替代 prompt 约束
  做法：把链接拼接、候选切片、ID 合法性校验、关键分支门禁等高风险动作下沉到 `if-else` 或 `code` 节点。
- 中间语义层与职责拆分
  做法：复杂链路优先拆成 `intent / slots / candidate_set / decision / response_plan` 或等价中间对象，避免一个节点同时做抽取、决策和呈现。
- Prompt 结构化与证据约束
  做法：优先使用“输入 / 决策规则 / 禁止项 / 输出格式 / 失败策略”的骨架，并明确缺证据时必须回退而不是脑补。
- 副作用与幂等性审查
  做法：识别所有外部调用、写状态和有副作用动作，写清是否可重试、重复执行影响、失败补偿或回滚策略。
- 资源边界与终止性审查
  做法：对循环、分页、聚合和长链路写清最大次数、超时、早停条件和输出大小上限。
- 回退契约一致性审查
  做法：确保主路径与 fallback 路径的输出字段、类型和语义一致；不一致时显式加归一化层。
- 敏感信息与最小暴露审查
  做法：检查 prompt、日志、错误信息、answer 和工具参数，只传必要字段，不把原始敏感片段直接透传。
- 依赖与版本假设审查
  做法：列出 DSL、插件、工具、字段写法和外部返回结构的版本假设，以及兼容风险。
- 变更影响面与回归差异审查
  做法：列出节点、selector、变量、输出契约、fallback、依赖和成本变化，并区分预期内 / 非预期 / 待确认变化。
- 观测契约审查
  做法：关键决策节点至少输出适用的 `reason_code`、`selected_path`、`evidence_ref`、`fallback_reason` 等内部观测字段。
- 能力块契约审查
  做法：每个能力块至少声明输入、输出、前置条件、副作用、失败语义、fallback、成本等级、决定性等级。
- 发布门禁审查
  做法：结构门、契约门、风险门、回归门、观测门和复核一致性门逐项给出通过 / 失败 / 待定 / 豁免状态。
- 内部语义对象最小 schema
  做法：为关键中间对象写最小结构、必填字段和语义说明，避免节点之间直接传大段自由文本。
- 输入归一化与证据快照
  做法：输入先统一清洗归一化；关键决策前冻结一份证据快照，后续生成只基于这份快照。
- 决定性边界审查
  做法：明确哪些逻辑必须规则化，哪些允许交给模型，例如关键标识校验、链接拼接、权限门禁、关键分支判定。
- 复核冲突仲裁
  做法：若多方复核结论不一致，输出冲突点、采纳结论、未采纳原因和仍待确认项。

### 有条件直接做

- 黄金样例与对抗回放
  条件：已有样例集、测试入口或可回放输入；满足条件时默认执行，否则在报告里列为优先补齐项。
- 置信度门与拒答门
  条件：任务存在明显误答风险，且可以定义低置信度时的澄清、回退或拒答策略。
- 运行期自修复
  条件：存在稳定可复用的轻量重试、备用解析链或安全 fallback。
- 能力裁剪
  条件：链路存在明显轻重两档，且可根据任务复杂度安全裁掉重组件。
- 输出形态切换
  条件：业务已明确需要 HTML、纯文本、结构化 JSON 或多语言等多种输出形态。

对上面这 5 项，报告里至少写清 3 件事：

- 条件是否满足
- 是否已执行 / 已设计
- 如果未执行，原因是什么

## 9. 默认值

- 默认策略档位
  `strict`
- 覆盖率通过最低标准
  适用路径覆盖率 `>= 95%`，且主路径、fallback 路径、外部异常路径不得缺测。
- 面向用户阅读的默认输出形态
  `纯文本`
- 面向 API 或 DSL 流程消费的默认输出形态
  `JSON`
- 可选输出形态
  `Markdown`、`HTML`

### 需要确认后做

- 时效字段口径
  需要确认哪些字段应视为时效信息，哪些允许当成稳定事实。
- 模板分层边界
  需要确认“通用骨架 / 业务策略层 / 能力块 / 文案层”如何切分，避免过度抽象。
- 能力注册表与场景包
  需要确认场景分类、能力块命名、拼装规则和复用边界。
- selector 别名层与适配层
  需要确认稳定变量命名、兼容范围和要吸收的外部差异。

默认直接做的项，除非用户明确反对，否则应直接纳入审查或建议。
有条件直接做的项，若条件不成立，要在报告里清楚标记原因。
需要确认后做的项，不要擅自定口径；列入“待确认项”即可。
