# 输出约定

默认先给人看的结论，再给结构化明细；不要一上来只吐长 YAML、长表格或裸状态码。

## 默认输出顺序

1. 任务概述
2. 模式判断
3. 节点清单
4. 连边清单
5. 字段检查表
6. 贯通性分析
7. 优化 / 修复建议
8. 校验摘要
9. 分级审查摘要
10. 最终结论

如果任务是只读审核，可以不输出 DSL 正文，但中间产物和结论顺序仍然成立。

如果任务是直接生成或改写 DSL：

- 先给中间产物，再给 DSL 正文
- 不要跳过结构分析，直接输出大段 YAML

## 校验摘要写法

每一轮检查都至少写清：

- 检查项
- 结论
- 这意味着什么
- 校验方式 / 依据

不要只写：

- `YAML_OK`
- `STRUCTURE_OK`
- `PY_CODE_OK`
- `PASS` / `FAIL`

如果用户明确要求机器可读标签，可以附在后面，但不能替代自然语言说明。

## 建议与结论写法

报告末尾必须区分：

- 已直接采纳的改进
- 有条件但未执行项
- 待确认项

建议项优先写成动作，而不是泛泛口号。例如：

- 补哪个字段
- 拆哪段职责
- 在哪个节点之间加归一化层
- 哪个门禁必须补证据

## 输出形态

- 面向用户阅读：默认 `纯文本`
- 面向 API 或 DSL 流程消费：默认 `JSON`
- 可选：`Markdown`、`HTML`

如果用户要求的输出形态超出上面范围，要明确说明是扩展约定，不要默默切换。

## 结论门槛

- 没做完 3 轮检查时，不要说“已精准校验”
- 没有阻塞项但仍有关键风险时，至少给“需要人工确认”
- 有阻塞项时，至少给“明显不完整”

完整报告格式需要时，使用 [report-template.md](../quality/report-template.md)。
